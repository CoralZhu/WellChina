# 1. 产品愿景 / Product Vision

## 1.1 一句话定位

**WellChina(栖康)是一个连接全球老年人与中国康养医疗机构的双边服务平台。**

我们不直接提供医疗服务,而是通过平台撮合的方式,让专业机构(医院、中医馆、康养中心)在线为海外老人提供从咨询、预订到陪诊、随访的一站式服务。

## 1.2 我们要解决什么问题

当一位外国老人想来中国接受医疗或康养服务时,他/她面临:

- 🌐 **信息不对称** —— 不知道哪家医院靠谱、哪家中医馆专业、价格是多少
- 🗣️ **语言障碍** —— 99% 的中国医院网站没有外文版本,医院现场也缺少跨语言支持
- 🛂 **流程复杂** —— 签证、机票、住宿、医院预约,要找十几个不同的供应商
- 🤝 **信任缺失** —— 没有评分机制,跨境医疗决策金额大(几千到几万美金),用户不敢决策

我们用**一个 App** 把这些问题一站式解决。

## 1.3 商业模式

**双边平台 + 飞猪式增值服务模型**:

| 端 | 谁 | 收入来源 |
|---|---|---------|
| **C 端** | 海外老年用户 | App 免费使用;按需购买核心医疗服务 + 增值服务(签证 / 机酒 / 接送) |
| **B 端** | 加盟机构(医院、中医馆、康养中心) | 平台佣金 / 订阅入驻费 / 增值服务分成 |

**核心 take rate 来源**:借鉴飞猪平台模式,在**核心服务下单后**引导用户购买**增值服务**(签证办理、机票酒店、机场接送),作为平台主要利润点。这个设计在第 3 章 user stories 中详细展开。

## 1.4 核心壁垒

公司前期已积累 **国内 7000+ 养老/医疗机构数据**,这是双边平台冷启动最关键的供给侧资源。我们不是从零做起,这绕过了所有平台模式最大的"鸡生蛋"难题。

## 1.5 长期愿景 vs 早期聚焦

| 时间 | 形态 |
|------|------|
| **MVP(0-12 个月)** | 聚焦 1-2 个核心市场试水(待确认),验证商业模型 |
| **成长期(1-3 年)** | 扩展至俄罗斯、北美华人、欧洲三大主市场,完成商业模型验证 |
| **长期(3-5 年)** | 成为全球老年人来华康养市场的"Booking.com",并探索"中医出海"延伸业务 |

## 1.6 核心使用者

**直接 App 用户(C 端)**:
- **首要**:60-75 岁海外老人本人(直接操作 App)
- **次要**:海外华人子女(协助决策与操作)

**后台用户(B 端)**:
- 加盟机构的运营人员(医院、康养中心、中医馆等)

## ⚠️ 第 1 章 待与业务方确认的开放问题

# 2. 用户画像 / User Personas

## 2.1 画像策略

我们识别出 **3 个核心用户画像**,代表 WellChina 在不同市场的典型用户。

**首要画像(Primary Persona)** 是 **Anna Volkova(俄罗斯)** —— 产品设计决策时,我们以服务好 Anna 为优先。原因有三:

1. 她的医疗需求最刚性(俄罗斯医疗等待 6 个月以上)
2. 所在市场竞争最少(欧美中介看不上俄语市场,英文医疗中介不服务)
3. 中俄医疗走廊已有政策基础(博鳌乐城 / 黑河边境 / 一带一路医疗走廊)

**次要画像** 是 Robert Chen(美华人)与 Margaret Williams(英国白人),代表华人市场基本盘与欧美增长曲线。

## 2.2 首要画像:Anna Volkova ⭐

### 基本信息

| 维度 | 内容 |
|-----|------|
| **姓名** | Anna Volkova(安娜·沃尔科娃) |
| **年龄** | 72 岁 |
| **居住地** | 莫斯科(俄罗斯) |
| **职业背景** | 退休内科医生 |
| **家庭状况** | 丈夫 5 年前去世,独居;女儿在英国伦敦工作,一年回来 1-2 次 |
| **数字水平** | 中等 —— 用 WhatsApp、VK、Yandex 没问题,但不会用海外 App |
| **语言** | 俄语为母语,英语能读不能说,完全不懂中文 |
| **典型客单价** | $3,000 - $8,000(治疗 + 21 天住宿 + 陪诊) |

### 核心痛点

1. **🩺 医疗等待时间长** —— 莫斯科约关节专科要等 6 个月以上,膝盖已痛 3 年
2. **💸 私立医院太贵** —— 俄罗斯私立手术 50-80 万卢布,且术后康复服务有限
3. **🌿 对中医好奇** —— 从女儿听说中国针灸效果好,YouTube 看过 BBC《中医纪录片》
4. **🤝 需要被陪伴** —— 没家人陪去中国,需要从机场接送到病房陪护的全套服务
5. **🛂 不懂跨境流程** —— 不知道办什么签证、能否带处方药、医保能不能报销

### Anna 心里的潜台词

> *"我已经一个人独自做了很多艰难的决定。这次去中国治病,如果能让我感觉**像有家人陪着**,而不是**像在办手续**,我就会信任你们。"*

### 设计这个 App 时,我们要为 Anna 做的关键决定

- ✅ **俄语必须 Day 1 就支持**(不是"以后再加")
- ✅ **价格要透明**(她要做财务决策,不要藏)
- ✅ **真人陪诊不能砍**(她最怕"被丢在异国")
- ✅ **支付方式必须支持卢布或欧元**(信用卡可能不通)
- ✅ **字体要大、操作要简**(72 岁的眼睛和反应)

# 3. 核心场景 / User Stories

## 3.1 用户旅程总览

我们将用户与 WellChina 的全部互动拆为 **5 个阶段**,覆盖一个海外老人从"听说我们" → "决定使用" → "完成治疗" → "回到本国后维持关系" 的完整生命周期。

```
阶段 1:发现 (Discovery)         → 怎么让 Anna 第一次知道 WellChina?
阶段 2:探索 (Exploration)       → 她怎么找到适合自己的服务?
阶段 3:决策 (Decision)          → 她怎么从"看看"变成"下单"?
阶段 4:履约 (Service Delivery)  → 她在中国期间,App 怎么陪伴她?
阶段 5:留存 (Retention)         → 她回家后,App 怎么继续连接?
```

每个阶段的具体 user stories 在下面分章列出,**每个 story 都标注 MVP 优先级**:

- **P0** = MVP 必备,不做产品就不成立
- **P1** = 重要但能晚点做,Phase 2 上线
- **P2** = 加分项 / 战略延伸,Year 2+ 考虑

---

## 3.2 阶段 1:发现 (Discovery)

### 设计原则

我们的获客策略覆盖三种渠道:**主动获客(社媒投放)+ 病毒传播(用户分享)+ 被动获客(搜索流量)**。三条腿走路,不依赖任何单一渠道。

### User Stories

| ID | User Story | 优先级 |
|---|-----------|-------|
| **S1.1** | 作为 Anna,我希望在社交媒体(Instagram / VK)看到博主推荐 WellChina 的内容,**能直接点链接下载**,以便快速进入 App 体验。 | **P0** |
| **S1.3** | 作为 Anna 的朋友,我希望能**把 WellChina 上某个具体服务页面分享给 Anna**(微信 / Telegram / WhatsApp),让她不用自己搜索就能直达。 | **P0** |
| **S1.5** | 作为 Anna,我希望在 Apple App Store 搜索 "healthcare China" 等关键词时**能找到 WellChina**,以便建立品牌感知。 | **P0** |

### 实现要点

- **S1.1**:配套需要营销落地页(Landing Page)和 App 下载链接的 deeplink 处理
- **S1.3**:每个服务详情页都需要原生分享按钮,生成带跳转的链接
- **S1.5**:需要做 ASO(App Store Optimization)优化,关键词:"China healthcare", "TCM", "Chinese medicine travel" 等

---

## 3.3 阶段 2:探索 (Exploration)

### 设计原则

**第一次打开 App 的 30 秒决定 70% 用户去留**(行业数据)。我们的策略:**最小注册门槛 + 多入口发现 + 强信任建立**。

### User Stories — 首次引导

| ID | User Story | 优先级 |
|---|-----------|-------|
| **S2.1** | 作为 Anna,我希望首次打开 App 时**能立刻选择我的语言**(俄语 / 英语 / 中文),否则我看不懂界面也用不下去。 | **P0** |
| **S2.3** | 作为 Anna,我希望看到 3 屏简洁的引导页,**了解 WellChina 能帮我做什么**,以便建立基本认知。 | **P0** |
| **S2.4** | 作为 Anna,我希望**不用注册账号就能浏览服务**(Guest Mode),以便降低我的决策门槛 —— 我还没确定要用,凭什么先注册? | **P0** |

### User Stories — 首页发现

> 用户进入首页后,有 5 条平行的服务发现路径,适配不同决策风格的用户:

| ID | User Story | 优先级 |
|---|-----------|-------|
| **S2.7** | 作为 Anna,我希望按"服务类型"浏览(中医 / 西医 / 康养旅居 / 陪诊),以便对应我的需求类别。 | **P0** |
| **S2.8** | 作为 Anna,我希望按"目的地"浏览(三亚 / 北京 / 上海 / 博鳌 ...),以便结合旅行偏好选择城市。 | **P0** |
| **S2.9** | 作为 Anna,我希望**直接搜索我的病症**(膝关节痛 / 心脏问题 / 糖尿病 ...),App 推荐合适机构 —— 我不会想"中医服务",我会想"我膝盖痛"。 | **P0** |
| **S2.10** | 作为 Anna,我希望打开**中国地图**直观看到各城市提供什么服务,以便建立空间认知。 | **P0** |
| **S2.11** | 作为 Anna,我希望看到"热门推荐"或其他用户点赞最多的服务,**以同侪选择获得社会证明**。 | **P0** |

### User Stories — 信任建立

> 用户找到感兴趣的服务后,需要看到足够信息才会进入决策。

| ID | User Story | 优先级 |
|---|-----------|-------|
| **S2.12** | 作为 Anna,我希望看到机构详情(医院介绍 / 医生团队 / 证书资质 / 三甲等级),以便判断是否专业。 | **P0** |
| **S2.13** | 作为 Anna,我希望看到具体服务详情(治疗套餐 / 包含内容 / 价格区间),以便预估我的支出。 | **P0** |
| **S2.16** | 作为 Anna,我希望看到"必读信息"(签证要求 / 处方药能否携带 / 跨境医保报销 ...),这些**实操问题决定我敢不敢来**。 | **P0** |
| **S2.14** | 作为 Anna,我希望看到**其他俄罗斯老人的真实评价(俄语)**,以便获得最强的同胞背书。 | **P1** |

### 实现要点

- **S2.9 病症搜索**:MVP 阶段不需要真做 AI,**先做 20-30 个高频病症的关键词映射**(后端写一张映射表,关键词 → 推荐机构 ID 列表)
- **S2.10 地图**:可以用高德/Mapbox + 自定义标记
- **S2.11 热门推荐**:早期数据不足时,可以先做"运营推荐"(运营手工选 20 个);用户量起来后切换到算法推荐

## 3.4 阶段 3:决策 (Decision)

### 设计原则

跨境医疗决策金额 $3k-30k,**用户从"看看"到"下单"的转化是产品最关键的漏斗**。设计核心:**多渠道答疑 + 透明信任 + 飞猪式增值**。

### User Stories — 解决疑问

| ID | User Story | 优先级 |
|---|-----------|-------|
| **S3.1** | 作为 Anna,我希望跟 **AI 文字助手**对话(俄语,24h 在线),问关于服务的细节问题,以便不必等待人工就能即时获得答复。 | **P0** |
| **S3.4** | 作为 Anna,我希望查看**其他用户问过的常见问题(FAQ)**,大多数我的疑问别人也问过。 | **P0** |
| **S3.3** | 作为 Anna(她是退休医生),我希望**上传我的病历和检查报告**,让机构先评估是否适合接收我,以便不白跑。 | **P0** |
| **S3.5** | 作为 Anna,我希望能**导出"服务报价单 PDF"**,发给在英国的女儿一起决策 —— 跨境医疗是大决策,不能我一个人定。 | **P0** |

### User Stories — 信任与决策

| ID | User Story | 优先级 |
|---|-----------|-------|
| **S3.6** | 作为 Anna,我希望看到**多种支付方式**(信用卡 / 银联 / Yandex Pay / WeChat Pay 等),适配不同地区用户的支付习惯。 | **P0** |
| **S3.7** | 作为 Anna,我希望看到**清晰的退款政策**(出发前取消条款 / 到中国不适应的处置),以便降低决策风险。 | **P0** |
| **S3.8** | 作为 Anna,我希望看到**完全透明的费用明细**(医疗 + 住宿 + 接送 + 陪诊拆解,**没有隐藏费用**)。 | **P0** |
| **S3.9** | 作为 Anna,我希望看到**平台担保(医疗事故赔付责任)**,简单告诉我:出问题有人负责。 | **P0** |

### User Stories — 完成下单

| ID | User Story | 优先级 |
|---|-----------|-------|
| **S3.11** | 作为 Anna,我希望**一键下单**,平台后台自动联系机构帮我安排,不用我同时跟多个供应商打交道。 | **P0** |
| **S3.16** ⭐ | 作为 Anna,下单后我希望看到 App **推荐附加增值服务**(签证办理 / 机票酒店预订 / 机场接送),类似飞猪平台买完机票后引导接送机和酒店。**这些是另外的价格,但都在同一个 App 里完成。** | **P0** |
| **S3.15** | 作为 Anna,我希望**预付定金锁定名额**,到中国后再付剩下的,以便降低我的财务风险。 | **P1** |
| **S3.13** | 作为 Anna,我希望选择"机票 + 酒店 + 机场接送"的**打包预订**,一次搞定所有。 | **P1** |

### ⭐ 关键产品策略亮点:**飞猪式 Cross-Sell 模型**(S3.16)

借鉴飞猪平台的模式:**核心服务保障基础体验,增值服务是平台主要 take rate**。

- **核心订单(医疗服务)** → 平台只赚轻佣金(5-10%)
- **增值服务(签证/机酒/接送)** → 平台主要利润来源(15-30% 毛利)

这与"分层服务模型"(在阶段 4 详述)形成完整商业闭环:**用户花的多 → 平台服务越深 → 体验越好 → 转介绍越多**。

### 实现要点

- **S3.1 AI 助手**:接入 Claude / GPT-4 + 俄语翻译。MVP 阶段先做 FAQ 模板对答,Phase 2 接入大模型对话
- **S3.3 上传病历**:涉及医疗数据合规(俄罗斯个人数据法 + 中国《数据安全法》),需要法务介入
- **S3.16 飞猪式 Cross-Sell**:需要对接签证服务商、OTA(携程/飞猪 API)、本地接送机供应商

## 3.5 阶段 4:履约 (Service Delivery)

### 设计原则

**线下做不好,品牌就崩了。** 这是产品最考验"线上线下闭环能力"的阶段。

设计核心:**分层服务模型(Tiered Service Model)** —— Anna 选了什么档位,App 提供对应深度的服务。

### ⭐ 关键产品策略亮点:**分层服务模型(Tiered Service Model)**

不是所有用户都付得起或愿意付陪诊费。**App 应该既能服务"全套陪诊"用户,也能服务"自己来"用户**:

| 服务档 | 用户类型 | 履约阶段获得的服务 |
|-------|---------|-------------------|
| **Tier 1 - 全套陪诊** | 已订陪诊 / 高客单 | 接机人身份验证、全程行程托管、人工关心电话、专属医院导航 |
| **Tier 2 - 基础保障** | 只订医疗 / 自主型 | 自助地图导航、AI 翻译、SOS 紧急按钮、必要时人工介入 |

这套模型让平台**对所有付费档位都能交付价值**,同时为升级到 Tier 1 留出明显的体验差距 —— **形成自然的 cross-sell 拉力**。

### User Stories — 落地机场

| ID | User Story | 优先级 | 服务层级 |
|---|-----------|-------|---------|
| **S4.1** | 作为 Anna,在机上我希望接到推送:"接机人身份证 + 联系方式",以便防止假司机骗局。 | **P0** | Tier 1 |
| **S4.2** | 作为 Anna,我希望**一键查看今天的完整行程**(几点接机 / 送到哪个酒店 / 明天几点去医院),以便我心里有数。 | **P0** | Tier 1 |
| **S4.5** | 作为 Anna(只订了医疗),我希望直接打开**外部地图(Apple Maps / Google Maps)** 自助导航到医院,以便不被强制留在 App 内。 | **P0** | Tier 2 |
| **S4.4** | 作为 Anna(任何档位),我希望随时一键 **SOS 紧急按钮**(迷路 / 身体不适 / 突发状况),24h 兜底。 | **P0** | All |

### User Stories — 医院现场

| ID | User Story | 优先级 |
|---|-----------|-------|
| **S4.6** | 作为 Anna,在医院里 App 是**我与医生的实时翻译器**(语音输入 → 中文转译给医生)。 | **P0** |
| **S4.7** | 作为 Anna,我希望**录音整个就诊过程,App 自动给出俄语总结**,以便事后回顾医嘱(口译可能漏信息)。 | **P0** |
| **S4.8** | 作为 Anna,我希望**拍药盒 / 处方,App 识别后给出服药提醒**,以便不混淆中文标签的药物。 | **P0** |
| **S4.9** | 作为 Anna,我希望在大医院里**查看院内位置导航**(护士台 / 化验室 / 药房),不至于在医院迷路。 | **P0** |
| **S4.10** | 作为 Anna(她是医生!),我希望**在 App 里有"个人健康档案"**(上传所有检查报告 / 医生评论),回俄罗斯后能继续追踪。 | **P0** |

### User Stories — 长居期间

| ID | User Story | 优先级 |
|---|-----------|-------|
| **S4.15** | 作为 Anna,我希望**有问题随时点 SOS 或在线聊**(俄语 / 24h),不至于在异国孤立无援。 | **P0** |
| **S4.11** | 作为 Anna,我希望**每日推送"今天的健康提醒"**(吃药 / 复诊 / 饮食建议),以便不漏过任何医嘱。 | **P0** |

### 实现要点

- **S4.6 实时翻译**:接入专业医疗翻译模型(Microsoft Translator + 医疗术语词库)
- **S4.7 录音 + 总结**:Whisper API + LLM 做总结,需要处理隐私合规(用户授权 + 加密存储)
- **S4.8 拍药 OCR**:可以用 Google Vision / 百度 OCR + 自建药品数据库

## 3.6 阶段 5:留存 (Retention)

### 设计原则

**一次性消费 → 长期关系是 LTV 的关键。** 设计核心:**功能性留存(随访 / 复购)+ 情感性留存(社群 / UGC)+ 增长性留存(推荐 / 代言)**。

### User Stories — 保持连接

| ID | User Story | 优先级 |
|---|-----------|-------|
| **S5.2** | 作为 Anna,我希望**远程随访**(以前的主治医生每月 1-2 次视频问诊,平台代纪记预约 + 翻译),这是我当时下单的关键理由之一。 | **P0** |
| **S5.5** | 作为 Anna,我希望**写评价 / 分享给未来考虑同医院的人**,这既帮助同胞,也让我作为医生的判断被看见。 | **P0** |
| **S5.4** | 作为 Anna,我希望进入**俄罗斯同胞老人聊天群**(同治马同病 / 同医院),病友是治疗期间最强的情感连接。 | **P1** |

### User Stories — 品牌代言

| ID | User Story | 优先级 |
|---|-----------|-------|
| **S5.6** | 作为 Anna,**下次来中国享受 VIP 折扣**(隐藏顶级费用 / 免预付定金),以便复购更顺畅。 | **P0** |
| **S5.7** | 作为 Anna,**推荐 1 位俄罗斯朋友下单后,获得下次预订 +$200 抵扣**,这是直接的推荐激励。 | **P0** |
| **S5.8** | 作为 Anna(她是医生 + 退休内科医生),我希望被邀请成为 **WellChina 俄罗斯区医生博主**,平台付费让我写专业文章 / 拍视频。 | **P1** |

### ⭐ 关键产品策略亮点:**UGC 留存飞轮**

S5.5(写评价)+ S5.4(同胞社群)+ S5.7(推荐有奖)+ S5.8(医生 KOL 计划)= **完整的 UGC 飞轮**:

```
Anna 治疗体验好
  ↓
她写评价(S5.5) → 直接驱动 S2.14(其他俄罗斯人看评价)
  ↓
她加入同胞群(S5.4) → 群里讨论 → 自然推荐
  ↓
她推荐朋友(S5.7) → 朋友下单 → 她得奖励
  ↓
她成为 KOL(S5.8) → 内容传播 → 驱动 S1.1(社媒发现)
```

这就是为什么 **Anna 一个用户,能带来 5-10 个新用户**。早期 LTV 极高。
